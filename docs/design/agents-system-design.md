# dAnalyzer Agents 系统详细设计

## 1. 设计哲学

### 1.1 核心哲学

dAnalyzer Agents 系统的核心哲学是**「按需调度 + 自主决策」**。

| 哲学 | 描述 | 实践方式 |
|------|------|----------|
| **单一入口** | 所有请求必须经过核心调度器 | danalyzer-core 是唯一执行入口 |
| **按需加载** | 仅在需要时加载完整指令 | 元数据预加载 + SKILL.md 延迟读取 |
| **自主决策** | Agent 自主决定执行路径 | 不预设固定流程，动态组合 |
| **无状态设计** | 每次请求独立处理 | 不依赖 workflows/ 或 storage/ |
| **事件驱动** | 通过 Hooks 实现自动化 | SessionStart/PreToolUse/PostToolUse/Stop |

### 1.2 设计意图

```
传统工作流引擎:
  用户 → 预定义流程 → 顺序执行 → 返回结果
  问题: 流程固化，扩展困难，不够灵活

dAnalyzer Agents:
  用户 → 核心调度器 → 理解需求 → 自主决策 → 按需执行 → 返回结果
  优势: 动态决策，按需加载，灵活扩展
```

### 1.3 与 Claude Code 的融合

dAnalyzer Agents 是专为 **Claude Code** 设计的智能体系统，充分利用 Claude Code 的独特能力：

| Claude Code 特性 | dAnalyzer 融合方式 |
|------------------|-------------------|
| Skill 工具 | 技能执行单元 |
| Hooks 系统 | 自动化协作 |
| Agent 工具 | 多 Agent 协调 |
| 全栈工具能力 | 数据分析全流程 |

## 2. 设计原则与理念

### 2.1 核心设计原则

#### 原则 1: 唯一入口原则

```
✅ 正确:
  用户输入 → danalyzer-core → Skills → result-formatter → 返回

❌ 错误:
  用户输入 → 直接调用 Skills (绕过 danalyzer-core)
```

**目的**: 确保所有请求都经过统一的调度和管控

#### 原则 2: 条件触发原则

```
需求模糊? ──是──→ demand-parse
           │
           └──否──→ 需求明确

任务复杂? ──是──→ task-planner
           │
           └──否──→ 任务简单
```

**目的**: 避免不必要的开销，仅在需要时调用辅助 Agent

#### 原则 3: 按需加载原则

```
SessionStart:
  ├─ 预加载 Skills 元数据 (~1KB)
  └─ 加载索引信息 (~100B)

Skill 执行时:
  └─ 按需读取 SKILL.md (~5-10KB)
```

**目的**: 控制上下文膨胀，保持高效

#### 原则 4: 安全嵌入原则

```
任何输出技能:
  data-query → data-analysis → visual/report/dashboard → security → result-formatter
                                                           ↑
                                                    脱敏 + 合规 (强制)
```

**目的**: 确保所有输出都经过安全处理

#### 原则 5: 错误容忍原则

```
错误分类:
  ├─ 可恢复错误 (超时/网络) → 重试/跳过
  ├─ 警告 (次要校验失败) → 记录警告，继续
  └─ 致命错误 (权限/合规) → 中止任务
```

**目的**: 提高系统鲁棒性

### 2.2 设计理念对比

| 维度 | 传统设计 | dAnalyzer 设计 |
|------|----------|----------------|
| 流程定义 | 预定义 XML/JSON | 运行时动态决策 |
| 技能调用 | 固定调用链 | 按需组合 |
| 上下文加载 | 全量加载 | 按需加载 |
| 错误处理 | 预定义分支 | Agent 智能决策 |
| 状态管理 | 需要持久化 | 无状态 |
| 扩展方式 | 修改流程定义 | 增加 Skill 即可 |

## 3. Agents 组件设计

### 3.1 Agents 目录结构

```
agents/
├── danalyzer-core.md      # ⭐ 核心调度器 (唯一入口)
├── demand-parse.md        # 需求拆解 Agent
├── task-planner.md        # 任务规划 Agent
├── data-validator.md      # 数据校验 Agent
├── error-handler.md       # 异常处理 Agent
├── result-formatter.md    # 结果格式化 Agent
└── template/              # 报告模板 (5个)
    ├── ad-hoc-analysis-template.md
    ├── data-export-template.md
    ├── indicator-monitor-template.md
    ├── monthly-report-template.md
    └── weekly-report-template.md
```

### 3.2 Agent 职责矩阵

| Agent | 类型 | 职责 | 触发条件 | 输出 |
|-------|------|------|----------|------|
| danalyzer-core | 核心 | 唯一入口，决策调度 | 始终 | 执行结果 |
| demand-parse | 辅助 | 需求拆解，明确目标 | 需求模糊 | 任务清单 |
| task-planner | 辅助 | 任务规划，生成计划 | 任务复杂 | 执行计划 |
| data-validator | 辅助 | 数据校验，质量验证 | 需要验证数据 | 校验报告 |
| error-handler | 辅助 | 异常处理，策略决策 | 发生错误 | 处理决策 |
| result-formatter | 辅助 | 结果格式化，标准化输出 | 需要输出 | 格式化结果 |

### 3.3 Agent 详细设计

#### 3.3.1 danalyzer-core (核心调度器)

**定位**: 整个系统的唯一执行入口

**核心职责**:
- 理解用户数据分析需求
- 决策技能组合
- 按需加载 Skills
- 协调执行流程
- 处理结果返回

**关键能力**:
```
1. 需求理解 (内嵌)
   - 解析数据源
   - 解析分析目标
   - 解析输出格式
   - 解析特殊要求

2. 技能决策 (内嵌)
   - 识别需求类型
   - 选择技能组合
   - 确定执行顺序

3. 按需加载
   - 仅在需要时读取 SKILL.md
   - 元数据预加载机制

4. 安全嵌入
   - 输出强制经过 security
   - 脱敏 + 合规检查
```

**配置**:
```yaml
name: danalyzer-core
description: 数据分析核心 Agent，负责理解需求并按需调用技能
tools: ["Read", "Write", "Edit", "Bash", "Glob", "Grep", "Agent", "Skill"]
model: sonnet
```

#### 3.3.2 demand-parse (需求拆解)

**定位**: 需求模糊时的智能拆解

**核心职责**:
- 拆解模糊业务需求
- 转化为可执行任务
- 明确目标、维度、口径边界
- 模糊需求自动追问

**触发条件**:
- 需求模糊 (无具体维度)
- 需求有多重含义
- 缺少关键约束 (时间范围、指标定义)

**输入输出**:
```json
// 输入
{
  "user_input": "分析销售数据"
}

// 输出
{
  "tasks": [
    {"id": 1, "description": "查询销售数据", "skill": "data-query"},
    {"id": 2, "description": "数据清洗", "skill": "data-clean"},
    {"id": 3, "description": "销售分析", "skill": "data-analysis"}
  ],
  "clarifications": [
    {"question": "需要分析哪个时间范围?", "options": ["近7天", "近30天", "自定义"]},
    {"question": "核心指标?", "options": ["销售额", "订单量", "客单价"]},
    {"question": "输出形式?", "options": ["图表", "报告", "数据文件"]}
  ]
}
```

#### 3.3.3 task-planner (任务规划)

**定位**: 复杂任务的执行计划生成

**核心职责**:
- 任务拆解
- 依赖分析
- 资源评估
- 时间估算
- 计划生成

**触发条件**:
- 任务数量 > 2
- 任务间有依赖关系
- 需要估算执行时间

**输入输出**:
```json
// 输入
{
  "tasks": [...],
  "available_resources": {...},
  "constraints": {...}
}

// 输出
{
  "execution_plan": [
    {"step": 1, "task": "data-query", "skill": "data-query", "duration": "2min", "dependencies": []},
    {"step": 2, "task": "data-clean", "skill": "data-clean", "duration": "1min", "dependencies": [1]},
    {"step": 3, "task": "data-analysis", "skill": "data-analysis", "duration": "3min", "dependencies": [2]}
  ],
  "total_duration": "6min",
  "parallel_groups": [[1], [2], [3]]
}
```

#### 3.3.4 data-validator (数据校验)

**定位**: 数据质量保障

**核心职责**:
- 格式校验
- 类型校验
- 范围校验
- 完整性校验
- 一致性校验

**校验能力**:
| 校验类型 | 说明 | 示例 |
|----------|------|------|
| 格式校验 | 检查数据格式 | 日期格式、邮箱格式 |
| 类型校验 | 检查字段类型 | 数值型、字符型 |
| 范围校验 | 检查数值范围 | 年龄 0-150 |
| 完整性校验 | 检查必填字段 | 订单号不能为空 |
| 一致性校验 | 检查关联数据 | 订单金额 = 商品金额之和 |

**输出**:
```json
{
  "status": "pass|fail|warning",
  "errors": [
    {"field": "amount", "type": "range", "message": "金额不能为负"}
  ],
  "warnings": [...],
  "report": "校验报告详情"
}
```

#### 3.3.5 error-handler (异常处理)

**定位**: 异常情况智能处理

**核心职责**:
- 错误分类
- 严重程度评估
- 处理策略决策
- 修复建议生成

**错误分类**:
| 错误类型 | 严重程度 | 示例 | 策略 |
|----------|----------|------|------|
| 取数超时 | 可恢复 | Hive查询超时 | 重试(3次) |
| 数据为空 | 警告 | 查询无结果 | 跳过 |
| 格式错误 | 可恢复 | CSV编码错误 | 重试(1次) |
| 权限不足 | 致命 | 无表权限 | 中止 |
| 规则违规 | 致命 | 敏感数据 | 中止 |

**处理决策**:
```json
{
  "decision": "retry|skip|degrade|abort",
  "max_retries": 3,
  "continue_execution": true,
  "repair_suggestion": "请检查网络连接后重试",
  "warning_message": "部分数据可能有误"
}
```

#### 3.3.6 result-formatter (结果格式化)

**定位**: 标准化输出

**核心职责**:
- 数据格式化
- 报告组装
- 模板填充
- 文件命名
- 元数据生成

**输出格式**:
| 格式 | 说明 |
|------|------|
| CSV | 逗号分隔值 |
| Excel | .xlsx 格式 |
| JSON | 结构化数据 |
| Markdown | 文档格式 |
| 图表 | 可视化图片 |

**输出结构**:
```json
{
  "success": true,
  "data": {...},
  "metadata": {
    "format": "csv",
    "filename": "sales_20260426.csv",
    "generated_at": "2026-04-26T10:00:00Z"
  },
  "errors": [],
  "warnings": []
}
```

```

  • 简单查询 → data-query

  • 需要清洗 → data-clean
  • 需要校验 → data-quality-check

  • 统计分析 → data-analysis
  • 用户分层 → rfm-analysis
  • 转化分析 → funnel-analysis
  • 高级建模 → model

  • 图表 → visual
  • 报告 → report
  • 看板 → dashboard
```

### 6.4 Hooks 集成

通过 hooks.json 配置实现自动化:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup",
        "hooks": [
          { "command": "hooks/run-hook.cmd session-start" },
          { "command": "learn/hooks/load-instincts" }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Skill",
        "hooks": [
          { "command": "hooks/run-hook.cmd post-skill-check" }
        ]
      },
      {
        "matcher": "Skill|Read",
        "hooks": [
          { "command": "learn/hooks/analyze-observe" },
          { "command": "learn/hooks/instinct-apply" }
        ]
      }
    ]
  }
}
```

### 6.5 工作流程

```
Claude Code 会话启动
    │
    ▼
SessionStart Hooks 执行
    ├─ session-start: 加载知识体系
    └─ load-instincts: 加载索引
    │
    ▼
用户输入
    │
    ▼
danalyzer-core 处理
    ├─ 需求理解
    ├─ 条件触发 (demand-parse/task-planner)
    ├─ 技能决策
    ├─ 按需加载 Skills
    └─ 执行
    │
    ▼
PostToolUse Hooks 执行
    ├─ analyze-observe: 记录行为
    ├─ instinct-apply: 智能建议
    └─ post-skill-check: 安全检查
    │
    ▼
返回结果
    │
    ▼
Stop Hooks 执行
    ├─ session-summary: 会话总结
    └─ pattern-detection: 模式检测
```

## 7. 方案优势与劣势

### 7.1 方案优势

| 优势 | 说明 | 效果 |
|------|------|------|
| **灵活扩展** | 增加 Skill 即可扩展功能 | 新增分析类型无需修改核心 |
| **按需加载** | 延迟加载完整指令 | 控制上下文膨胀 |
| **自主决策** | Agent 动态决定执行路径 | 适配多场景需求 |
| **条件触发** | 满足条件才调用辅助 Agent | 避免不必要开销 |
| **安全嵌入** | 输出强制经过 security | 确保数据安全 |
| **无状态设计** | 不依赖持久化 | 简化部署和维护 |
| **事件驱动** | Hooks 实现自动化 | 增强系统能力 |
| **多 Agent 协作** | 按需调用辅助 Agents | 复杂任务处理 |

### 7.2 方案劣势

| 劣势 | 说明 | 缓解措施 |
|------|------|----------|
| **调试困难** | 动态决策难以追踪 | 增加日志和追踪 |
| **不确定性** | Agent 决策可能不稳定 | 定义决策规则 |
| **上下文限制** | 受限于 Claude Code 上下文 | 按需加载控制大小 |
| **依赖模型能力** | 依赖 LLM 理解能力 | 优化提示词设计 |
| **错误传播** | 链式调用可能放大错误 | 完善的异常处理 |

### 7.3 适用场景

| 场景 | 适用性 | 原因 |
|------|--------|------|
| 简单数据查询 | ✅ 最佳 | 快速响应，按需加载 |
| 复杂分析任务 | ✅ 良好 | 多 Agent 协作 |
| 标准化报告 | ✅ 良好 | 模板化输出 |
| 实时决策 | ⚠️ 慎重 | 依赖 LLM 响应时间 |
| 高频调用 | ⚠️ 慎重 | 上下文开销 |

## 8. 设计最佳实践

### 8.1 Agent 设计原则

1. **单一职责**: 每个 Agent 只负责一个明确的功能
2. **清晰接口**: 定义明确的输入输出格式
3. **条件触发**: 不预设固定调用，按需触发
4. **错误处理**: 每个 Agent 都要有错误处理逻辑
5. **日志输出**: 记录关键决策和执行状态

### 8.2 技能组合原则

1. **安全优先**: 所有输出技能必须嵌入 security
2. **按需加载**: 只加载需要的 SKILL.md
3. **灵活组合**: 根据需求动态调整技能组合
4. **依赖管理**: 明确技能间的依赖关系

### 8.3 异常处理原则

1. **分类处理**: 根据错误类型采取不同策略
2. **降级机制**: 关键错误提供降级方案
3. **错误传播**: 正确传递错误信息
4. **日志记录**: 记录错误详情便于排查

### 8.4 性能优化原则

1. **上下文控制**: 全程控制在 ~2KB 以内
2. **按需加载**: 仅加载必要的指令和数据
3. **缓存策略**: 索引信息缓存，实时数据按需获取
4. **并行执行**: 独立任务可并行执行
