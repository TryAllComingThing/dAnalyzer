## 1 设计哲学

### 1.1 核心设计理念

Rules 与 Checks 系统采用**分层防护 + 纵深防御**的设计哲学，通过多层级、渐进式的校验机制，确保数据分析任务在符合规范的前提下安全执行。

```
┌─────────────────────────────────────────────────────────────────┐
│                        安全防护金字塔                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                        ┌─────────┐                              │
│                        │  Legal  │  ← 最高优先级，零容忍        │
│                        │  (法律) │     违规即终止               │
│                        └────┬────┘                              │
│                        ┌────┴────┐                              │
│                        │  Core   │  ← 企业核心，一致性强制      │
│                        │  (企业) │     不一致禁止执行            │
│                        └────┬────┘                              │
│                        ┌────┴────┐                              │
│                        │  Base   │  ← 基础规范，提示式引导      │
│                        │  (基础) │     不强制但给出优化建议    │
│                        └────┬────┘                              │
│                        ┌────┴────┐                              │
│                        │ Dynamic │  ← 动态生效， временные     │
│                        │  (动态) │     临时合规/口径变更        │
│                        └─────────┘                              │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│  Checks 自动化钩子 (前置/后置双重保障)                           │
│  ┌──────────────┬──────────────┬──────────────┐                │
│  │ Data Quality │   Caliber    │  Compliance  │                │
│  │  数据质量    │    口径      │    合规      │                │
│  └──────────────┴──────────────┴──────────────┘                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 分层设计的哲学基础

| 层级 | 设计原则 | 哲学依据 |
|------|----------|----------|
| Legal | **零容忍原则** | 法律合规是不可逾越的红线，采用拦截式校验 |
| Core | **一致性原则** | 企业核心规范是业务稳定的基础，采用前置强校验 |
| Base | **引导性原则** | 基础规范是最佳实践，采用提醒式建议 |
| Dynamic | **时效性原则** | 临时规则有明确生命周期，采用条件触发 |

### 1.3 Rules 与 Checks 的定位差异

| 维度 | Rules (规则) | Checks (校验) |
|------|--------------|---------------|
| **本质** | 规范约束定义 | 自动化执行机制 |
| **粒度** | 业务规则层面 | 技术实现层面 |
| **触发** | CLI任务执行前 | 任务前后双触发 |
| **处理** | 定义"应该怎么做" | 验证"实际做得如何" |
| **优先级** | 绝对优先 | 辅助保障 |
| **执行方式** | 规则引擎解释执行 | 自动化脚本执行 |

## 2 设计原则与理念

### 2.1 分级管理原则

#### 2.1.1 Legal 级规则（法律级）

**设计理念**: 最高优先级，零容忍，拦截式校验

```
特征:
- 不可违背的法律红线
- 违规后立即终止任务
- 无例外豁免机制
- 需明确告知违规原因

涉及领域:
- 隐私保护 (privacy-protection.md)
- 数据安全 (data-security.md)
- 敏感数据处理 (sensitive-data.md)
- 出口管控 (export-control.md)
```

#### 2.1.2 Core 级规则（企业核心）

**设计理念**: 企业级核心规范，前置校验，一致性强制

```
特征:
- 企业业务稳定性基础
- 不一致时禁止执行
- 需要指标字典支撑
- 支持版本管理

涉及领域:
- 指标口径统一 (indicator-caliber.md)
- 维度标准规范 (dimension-standard.md)
- 命名规范 (naming-convention.md)
- 数据时效 (data-freshness.md)
```

#### 2.1.3 Base 级规则（基础规范）

**设计理念**: 编码/编写最佳实践，提醒式校验

```
特征:
- 提升代码质量
- 不强制终止执行
- 提供优化建议
- 可配置警告级别

涉及领域:
- SQL编写规范 (sql-write.md)
- 文件命名规范 (file-naming.md)
- 代码注释规范 (code-comment.md)
- 数据类型规范 (data-type.md)
- 报告格式规范 (report-format.md)
```

#### 2.1.4 Dynamic 级规则（动态规则）

**设计理念**: 临时合规/口径变更，时效性控制

```
特征:
- 明确的生效时间窗口
- 到期自动失效/归档
- 支持版本迭代
- 可覆盖基础规则

典型场景:
- 节假日统计口径 (holiday-caliber-2026.md)
- 促销期间口径 (promotion-caliber.md)
- 测试环境特殊规则 (test-environment-rule.md)
- 新指标试运行 (new-indicator-rule.md)
```

### 2.2 渐进式校验原则

```
任务执行流程中的校验层级:

CLI触发
    │
    ▼
┌─────────────────┐
│  Legal 校验     │  ← 第一道防线，拦截违规
│  (前置+后置)    │     任何违规立即终止
└────────┬────────┘
         │ 通过
         ▼
┌─────────────────┐
│  Core 校验     │  ← 第二道防线，前置校验
│  (前置)         │     不一致禁止执行
└────────┬────────┘
         │ 通过
         ▼
┌─────────────────┐
│  Base 校验     │  ← 第三道防线，提醒式
│  (前置)         │     给出优化建议
└────────┬────────┘
         │ 通过
         ▼
┌─────────────────┐
│ Dynamic 校验   │  ← 条件触发
│  (生效期内)    │     临时规则
└────────┬────────┘
         │ 通过
         ▼
    任务执行
         │
         ▼
┌─────────────────┐
│  Checks 后置    │  ← 输出质量保障
│  (结果校验)     │     防止问题输出
└─────────────────┘
```

### 2.3 可观测性原则

每个规则和校验都应产生可追溯的日志记录：

```
日志记录要求:
- 触发时间
- 校验类型
- 输入数据特征(脱敏后)
- 校验结果
- 处理措施
- 执行耗时
```

### 2.4 可配置原则

```
可配置项:
- 阈值参数 (空值比例、异常值边界等)
- 处理方式 (预警/终止/自动处理)
- 生效时间 (动态规则)
- 触发条件 (前置/后置)

配置优先级:
CLI参数 > 配置文件 > 默认值
```

### 3.3 规则定义格式

每个规则文件采用统一的 Markdown 格式：

```markdown
# 规则名称

## 核心职责
(简明扼要描述规则的用途)

## [核心内容]
(根据规则类型不同，内容有所差异)

## 校验时机
- 前置校验 / 后置校验
- 调用条件

## 处理方式
| 场景 | 处理方式 |
|------|----------|
| 违规A | 终止任务 |
| 违规B | 预警 |
```

```markdown
| 校验项 | 规则 | 处理方式 |
|--------|------|----------|
| ...    | ...  | ...      |

- 依赖的规则

## 输出结果
- 校验报告
- 异常列表
- 修正建议

## 依赖
- 依赖的资源
```

## 4 执行流程

### 4.1 规则校验执行流程

```
┌──────────────────────────────────────────────────────────────────┐
│                    Rules 校验执行流程                              │
└──────────────────────────────────────────────────────────────────┘

1. CLI触发任务
   │
   ▼
2. 加载 Legal 级规则
   │
   ├─ 读取 rules/legal/*.md
   ├─ 执行拦截式校验
   └─ 违规? ──是──→ 立即终止，输出违规详情
              │
              否
              ▼
3. 加载 Core 级规则
   │
   ├─ 读取 rules/core/*.md
   ├─ 读取指标字典资产
   ├─ 执行前置校验
   └─ 不一致? ──是──→ 禁止执行，提示不一致详情
              │
              否
              ▼
4. 加载 Base 级规则
   │
   ├─ 读取 rules/base/*.md
   ├─ 执行提醒式校验
   └─ 记录优化建议
              │
              ▼
5. 动态规则检查
   │
   ├─ 检查当前日期是否在生效期内
   ├─ 仅执行在有效期内的动态规则
   └─ 到期规则自动跳过
              │
              ▼
6. 记录校验日志
   │
   ├─ 记录所有校验结果
   ├─ 记录处理措施
   └─ 记录执行耗时
              │
              ▼
7. 任务执行 / 终止
```

### 4.3 Rules 与 Checks 协同流程

```
┌──────────────────────────────────────────────────────────────────┐
│               Rules 与 Checks 协同执行流程                         │
└──────────────────────────────────────────────────────────────────┘

     Rules (定义)                          Checks (执行)
        │                                     │
        │  ┌─────────────────┐                │
        │  │ Legal Rules    │                │
        │  │ - 隐私保护     │ ───────────────┼──────────→ Compliance Scan
        │  │ - 数据安全     │                │                │
        │  │ - 敏感数据     │                │                │
        │  └─────────────────┘                │
        │                                     │
        │  ┌─────────────────┐                │
        │  │ Core Rules      │                │
        │  │ - 指标口径     │ ───────────────┼──────────→ Caliber Check
        │  │ - 维度标准     │                │                │
        │  │ - 命名规范     │                │                │
        │  └─────────────────┘                │
        │                                     │
        │  ┌─────────────────┐                │
        │  │ Base Rules     │                │
        │  │ - SQL规范      │ ───────────────┼──────────→ Data Quality
        │  │ - 注释规范     │                │                │
        │  │ - 报告格式     │                │                │
        │  └─────────────────┘                │
        │                                     │
        │  ┌─────────────────┐                │
        │  │ Dynamic Rules  │                │
        │  │ - 节假日口径   │ ───────────────┼──────────→ 条件触发
        │  │ - 促销口径     │                │                │
        │  └─────────────────┘                │
        │                                     │
        └─────────────────────────────────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │   校验结果聚合   │
                    │   执行决策确认  │
                    └─────────────────┘

执行流程时序:

T0: CLI 触发任务
    │
T1: Rules 前置校验 (Legal → Core → Base → Dynamic)
    │  Legal: 拦截式
    │  Core: 前置校验
    │  Base: 提醒式
    │  Dynamic: 条件触发
    │
T2: Checks 前置校验 (Data Quality → Caliber → Compliance)
    │  前置: 输入验证
    │  可选: 实时监控
    │
T3: 任务执行
    │  数据查询 → 数据清洗 → 数据分析 → 可视化 → 报告生成
    │
T4: Checks 后置校验 (Data Quality → Caliber → Compliance)
    │  后置: 输出验证
    │  结果质量保证
    │
T5: Rules 后置校验 (Legal 敏感数据扫描)
    │  输出前最终检查
    │
T6: 输出结果 / 任务终止

时间线:
CLI ──T1(Rules)──T2(Checks前)──T3(执行)──T4(Checks后)──T5(Rules后)──结果
                   │              │           │            │
              Legal优先        质量保证    数据质量     敏感数据
              强制执行                      口径验证     最终扫描
```

## 5 模块协同

### 5.1 Rules 与 Skills 的协同

```
Rules 定义约束 ──────────→ Skills 执行任务
      │                         │
      │ 技能选择依据            │ 技能执行前校验
      │ 哪些技能需要触发        │ 是否符合规则要求
      │                         │
      ├─ data-query         ←──├─ Legal/Core 规则校验
      │  需符合隐私保护规则        通过后才能执行
      ├─ rfm-analysis       ←──├─ Core 口径规则校验
      │  需使用标准指标定义        转化率、活跃用户口径
      ├─ sql-write         ←──├─ Base 规范校验
      │  需遵循SQL编写规范         关键字大小写等
      └─ compliance        ←──├─ Legal 合规规则
           需通过合规扫描
```

### 5.2 Rules 与 Agents 的协同

```
danalyzer-core (核心调度器)
      │
      ├─→ 需求解析 ──→ 判断需要哪些规则
      │
      ├─→ 规则加载 ──→ 按优先级加载规则
      │    Legal > Core > Base > Dynamic
      │
      ├─→ 校验执行 ──→ 调用对应规则
      │
      └─→ 结果处理 ──→ 终止/预警/继续

其他 Agents:
- demand-parse: 需求拆解时考虑规则约束
- task-planner: 规划时引用规则作为约束条件
```

## 6 优劣势分析

### 6.1 设计优势

#### 6.1.1 分层防护机制

| 优势 | 说明 |
|------|------|
| **优先级明确** | Legal → Core → Base → Dynamic 优先级清晰，避免规则冲突 |
| **渐进式校验** | 越早发现问题成本越低，分层实现早发现早预防 |
| **零信任设计** | Legal 级采用零信任原则，任何违规立即终止 |
| **灵活性** | 动态规则支持时效性场景，无需修改核心逻辑 |

#### 6.1.2 双校验机制

| 优势 | 说明 |
|------|------|
| **前后呼应** | Rules 定义约束，Checks 自动化执行验证 |
| **质量保障** | 前置防止问题，后置确保输出质量 |
| **双重保险** | 关键场景双重校验，降低风险 |

#### 6.1.3 标准化设计

| 优势 | 说明 |
|------|------|
| **统一格式** | 所有规则/校验采用 Markdown 格式，易读易维护 |
| **结构清晰** | 统一的章节结构，核心职责/校验内容/处理方式 |
| **可扩展** | 易于添加新规则/校验，不影响现有结构 |

#### 6.1.4 与其他模块良好协同

| 协同模块 | 协同方式 |
|----------|----------|
| Skills | Skills 执行前需通过 Rules 校验 |
| Agents | Core Agent 统一调度规则加载与执行 |
| Industry Data | 口径校验依赖指标字典资产 |
| Learn | 规则违规模式可作为学习素材 |

### 6.2 设计挑战

#### 6.2.1 规则冲突风险

| 挑战 | 描述 |
|------|------|
| **跨级冲突** | 低优先级规则可能与高优先级冲突 |
| **动态规则冲突** | 动态规则与基础规则可能冲突 |
| **多规则叠加** | 多个规则同时触发，可能产生矛盾的处理方式 |

**缓解措施**:
- 优先级机制: Legal > Core > Base > Dynamic
- 冲突检测: 规则加载时进行冲突检查
- 明确处理: 冲突时按优先级选择

#### 6.2.2 维护成本

| 挑战 | 描述 |
|------|------|
| **规则同步** | 多处规则需要保持一致性 |
| **版本管理** | 规则版本更新需要追踪 |
| **过期规则** | 动态规则需要及时归档 |

**缓解措施**:
- 规则集中管理，避免重复定义
- 版本号显式标注
- 过期前30天自动提醒

#### 6.2.3 执行性能

| 挑战 | 描述 |
|------|------|
| **规则加载** | 多个规则文件需要加载解析 |
| **校验耗时** | 复杂校验可能影响任务执行时间 |
| **实时性要求** | 某些场景需要毫秒级响应 |

**缓解措施**:
- 规则元数据预加载
- 延迟加载: 仅加载需要的规则
- 缓存: 规则校验结果缓存

#### 6.2.4 配置复杂度

| 挑战 | 描述 |
|------|------|
| **阈值配置** | 多处阈值需要统一管理 |
| **开关控制** | 规则/校验启用/禁用需要统一控制 |
| **参数传递** | CLI 参数到规则系统的传递 |

**缓解措施**:
- 统一配置中心
- 配置优先级: CLI > 配置文件 > 默认值

### 6.3 与行业实践对比

| 维度 | 本设计 | 传统做法 |
|------|--------|----------|
| **规则优先级** | 四级分层 (Legal/Core/Base/Dynamic) | 二元 (强制/建议) |
| **动态规则** | 支持明确生效时间窗口 | 硬编码/注释 |
| **校验触发** | 前置+后置双触发 | 仅后置或手动 |
| **处理方式** | 终止/预警/自动处理 | 仅警告 |
| **标准化** | 统一 Markdown 格式 | 多样化/私有格式 |
| **可观测性** | 完整日志记录 | 较少 |

### 6.4 改进建议

| 优先级 | 建议 | 说明 |
|--------|------|------|
| **高** | 规则冲突检测机制 | 规则加载时自动检测冲突 |
| **高** | 规则版本管理 | 实现规则版本追踪和回滚 |
| **中** | 规则执行性能优化 | 实现规则缓存和延迟加载 |
| **中** | 规则编辑器 UI | 图形化规则管理界面 |
| **低** | 规则测试框架 | 规则逻辑自动化测试 |
