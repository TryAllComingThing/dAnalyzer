# 洞察生成 (Insight Generation)

## 概述

从分析结果中提炼可执行的业务洞察。四步框架：识别信号（趋势/异常/对比/结构）→ 评估可信度（高/中/低/待定）→ 生成业务建议（发现→含义→行动）→ 优先级排序（P0 紧急/P1 重要/P2 关注/P3 信息）。输出结构化洞察摘要，每个发现附带数据支撑和可信度标注。

## 何时使用

- **触发:** 分析或建模完成，需要提炼业务结论
- **触发:** 用户问"发现了什么"、"给些建议"、"该怎么办"
- **触发:** 报告需要洞察摘要章节
- **不要用于:** 原始分析执行（使用 data-analysis 或 model）
- **不要用于:** 纯可视化无洞察需求（使用 visual）
- **不要用于:** 分析结果尚未就绪的数据清洗/查询阶段

---

## 核心步骤

1. **接收分析结果** — 从 data-analysis 或 model 获取统计指标和效应量
2. **识别信号** — 扫描趋势信号/异常信号/对比信号/结构信号 → 详见「洞察提炼规则 > 1. 识别信号」
3. **评估可信度** — 高（n>1000）/ 中（100-1000）/ 低（<100）/ 待定（无样本量信息）→ 详见「洞察提炼规则 > 2. 评估可信度」
4. **生成业务建议** — 每个发现映射到：数据发现 → 业务含义 → 建议行动 → 详见「洞察提炼规则 > 3. 生成业务建议」
5. **优先级排序** — P0 紧急 / P1 重要 / P2 关注 / P3 信息 → 详见「洞察提炼规则 > 4. 优先级排序」
6. **输出洞察摘要** — Markdown 格式：核心发现 + 业务建议 + 关注事项 → 详见「输出格式」

---

## 洞察提炼规则

> 对应核心步骤第 2-5 步

分析结果到手后，按以下框架提炼洞察：

### 1. 识别信号

| 信号 | 触发条件 | 示例输出 |
|------|---------|---------|
| 趋势信号 | 连续 3 期同向变化，或环比 > 10% | "销售额连续 3 周增长，周均 +12%" |
| 异常信号 | 超出 IQR/3σ 范围，或业务合理范围 | "周二订单量骤降 75%，疑为系统延迟" |
| 对比信号 | 组间差异 > 20%，Cohen's d > 0.5 | "华东区域 ARPU 显著高于其他区域 (d=0.7)" |
| 结构信号 | 单一维度占比 > 40% | "电子产品贡献 45% 营收，集中度偏高" |

### 2. 评估可信度

仅当分析结果包含样本量信息时进行。如无样本量数据，标记可信度为"待定"。
- **高**: 样本充足 (>1000)，趋势一致，统计显著
- **中**: 样本一般 (100-1000)，趋势模糊，波动大
- **低**: 样本不足 (<100)，单一数据点，需更多数据验证

### 3. 生成业务建议

每个洞察必须附带可执行的业务建议：

```
数据发现 → 业务含义 → 建议行动
"退款率上升 5%" → "客服可能无法满足新增用户需求" → "建议加派人手并优化自助退款流程"
```

### 4. 优先级排序

| 优先级 | 条件 |
|--------|------|
| **P0-紧急** | 核心指标异常偏离 > 50%，需立即处理 |
| **P1-重要** | 影响 > 20% 用户/营收，24h 内处理 |
| **P2-关注** | 趋势信号，本周内跟进 |
| **P3-信息** | 一般性发现，记录备查 |

### 5. 复合信号识别

多个独立信号叠加时，不能仅分别报告单一信号，必须进行交叉分析以发现深层问题。

**复合信号类型：**

| 复合模式 | 触发条件 | 业务含义 | 处理方式 |
|---------|---------|---------|---------|
| **趋势 + 异常叠加** | 趋势信号触发的同时，某一期触发异常信号 | 趋势中的断点，可能是外部事件冲击或数据采集故障 | 标注断点时间，单独分析该期原因，排除数据故障后归因到业务事件 |
| **趋势 + 结构交叉** | 整体趋势变化 + 某一维度占比显著变化 | 趋势可能由结构变化驱动（如总销售额上升但实为单一品类暴增拉动） | 做贡献度分解：各维度对总趋势的贡献率，区分「真实增长」与「结构偏移」 |
| **对比 + 异常叠加** | 组间差异显著 + 某一组内部异常值频发 | 差异可能由该组的异常个案驱动，非系统性差异 | 剔除异常值后重新对比，分别报告含/不含异常值的结果 |
| **对比 + 结构交叉** | 组间差异显著 + 各组内部结构不同 | 差异可能由结构差异解释（如 A 区用户年龄偏大导致 ARPU 偏高） | 控制结构变量后再次对比（分层分析），标注是否需要更细粒度拆解 |
| **异常 + 结构交叉** | 异常信号触发 + 该异常集中在特定维度 | 异常非随机分布，指向特定环节或群体 | 交叉分析异常值的维度分布（按品类/渠道/区域），定位异常集中点 |
| **三信号叠加** | 趋势 + 异常 + 对比同时触发 | 系统性变化而非偶然波动，需全面诊断 | 升级为 P0，建议触发完整归因分析（model/references/attribution.md） |

**复合信号处理优先级：**
- 三信号叠加 > 趋势 + 异常 > 对比 + 异常 > 趋势 + 结构 > 对比 + 结构 > 异常 + 结构
- 高优先级复合信号触发时，单一信号的结果仅作为背景说明，不独立输出

## 输出格式

> 对应核心步骤第 6 步

```markdown
## 洞察摘要

### 核心发现
1. **[信号类型]** 发现描述 — 数据支撑 — 可信度
2. ...

### 业务建议
1. 建议行动 — 预期效果 — 优先级

### 关注事项
- ⚠️ 需关注的风险/异常
- ✅ 积极信号
```

## 常见借口与纠正

| 借口 | 现实 |
|--------|---------|
| "趋势很明显，不需要再评估可信度" | 无样本量标注的洞察可能误导决策；小样本的"显著趋势"可能只是噪声 |
| "给几个笼统的建议就行" | 无优先级和无预期效果的建议不可执行；每个建议必须具体且可验证 |
| "分析结果已经有结论了，洞察是多余的" | 分析结论 ≠ 业务洞察；分析说"GMV 下降 10%"，洞察说"因为这 10% 集中在华东，建议优先排查华东渠道" |

## 红线

- **严重性降级:** 核心指标偏离 >50% 却标为 P3（信息）— 必须为 P0（紧急）
- **虚假高可信度:** 无样本量信息却标为"高"可信度 — 改为"待定"
- **不可执行建议:** 业务建议缺少预期效果描述 — 无法落地执行
- **方向矛盾:** 洞察方向与数据矛盾（如洞察说"下降"但数据呈上升趋势）

## 验证

1. 验证信号覆盖：确认 4 类信号（趋势/异常/对比/结构）均已扫描
2. 验证可信度标注：确认每个洞察有可信度等级（高/中/低/待定）
3. 验证数据支撑：确认每个洞察包含具体数字或百分比作为证据
4. 验证建议可执行：确认每条建议包含预期效果描述
5. 验证优先级排序：确认 P0-P3 分配符合定义标准
6. 验证输出格式：确认输出匹配洞察摘要模板结构
