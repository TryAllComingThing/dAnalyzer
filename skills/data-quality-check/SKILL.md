# 数据质量校验 (Data Quality Check)

## 概述

六维度数据质量评分系统（完整性、一致性、准确性、时效性、唯一性、有效性），输出 0-100 加权总分及逐维度报告。定位为只读校验层 — 不修改数据，仅评估和标记质量问题。适用于数据清洗后验证、多源融合前把关、报告输出前可信度确认。

## 何时使用

- **触发:** 在 data-clean 执行后验证清洗结果
- **触发:** 用户要求"质量检查"、"数据质量"或"质量报告"
- **触发:** 多源融合前检查各源质量
- **触发:** 报告输出前验证数据可信度
- **不要用于:** 数据清洗本身（使用 data-clean）
- **不要用于:** 仅有汇总数据无明细行（样本不足）
- **不要用于:** 无结构化字段的非结构化文本

---

## 核心步骤

1. **确定检查范围** — 明确哪些表/字段纳入检查
2. **逐维度执行** — 按六大维度（完整性/一致性/准确性/时效性/唯一性/有效性）逐项检查并记录结果 → 详见「六大质量维度」
3. **计算得分** — 各维度按公式评分（0-100），加权求和得总分 → 详见「质量评分方法」
4. **生成质量报告** — 输出 JSON 报告，含总分、评级、逐维度明细、失败项列表 → 详见「输出格式」
5. **决策操作** — 根据评级决定：≥95 直接用 / 85-94 注意低分项 / 70-84 针对性修复 / <70 拒绝使用

---

## 六大质量维度

> 对应 核心步骤 第 2 步

### 1. 完整性 (Completeness)

检查数据是否缺失关键字段。

| 检查项 | 方法 | 通过标准 |
|--------|------|----------|
| 必填字段非空 | 逐字段 COUNT(NULL) | 空值率 < 1% |
| 关键字段完整 | 主键/关联键无 NULL | 0% 空值 |
| 时间序列连续 | 检查日期间隙 | 连续率 > 95% |

### 2. 一致性 (Consistency)

检查数据是否符合业务逻辑约束。

| 检查项 | 方法 | 通过标准 |
|--------|------|----------|
| 字段内格式一致 | 检查枚举值分布 | 异常值 < 5% |
| 跨字段逻辑一致 | 如 下单时间 ≤ 支付时间 | 违反率 < 1% |
| 参照完整性 | 外键在关联表存在 | 匹配率 > 99% |

### 3. 准确性 (Accuracy)

检查数据是否反映真实业务状态。

| 检查项 | 方法 | 通过标准 |
|--------|------|----------|
| 范围检查 | 数值在合理业务范围 | 超界 < 1% |
| 统计合理性 | 均值/中位数对比预期 | 偏差 < 3σ |
| 业务规则验证 | 如 退款额 ≤ 订单额 | 违反率 = 0% |

### 4. 时效性 (Timeliness)

检查数据是否在预期时间内到位。

| 检查项 | 方法 | 通过标准 |
|--------|------|----------|
| 数据新鲜度 | 检查最大时间戳 | 未超过 SLA 延迟 |
| 批处理准点 | 检查更新时间与预期 | 延迟 < 15min |
| 实时延迟 | 检查生产时间差 | 延迟 < 5min |

### 5. 唯一性 (Uniqueness)

检查是否存在意外重复。

| 检查项 | 方法 | 通过标准 |
|--------|------|----------|
| 主键唯一 | COUNT(DISTINCT) = COUNT(*) | 重复率 = 0% |
| 业务键唯一 | 按业务规则去重检查 | 重复率 < 1% |
| 部分重复 | 关键字段组查重 | 重复率 < 3% |

### 6. 有效性 (Validity)

检查数据是否符合预期格式和编码。

| 检查项 | 方法 | 通过标准 |
|--------|------|----------|
| 格式有效 | 正则匹配预期模式 | 有效 > 98% |
| 编码正确 | UTF-8 检测 | 无乱码 |
| 枚举值合法 | 值在允许集合内 | 合法率 = 100% |

## 复合检查项

| 检查 | 涉及维度 | 说明 |
|------|----------|------|
| 跨源一致性 | 一致性 + 准确性 | 同一实体在不同源中的值相同 |
| 时间戳合理性 | 时效性 + 准确性 | created_at ≤ updated_at ≤ now |
| 指标连贯性 | 完整性 + 一致性 | 如 总销售额 = 各渠道求和 |

## 执行流程

```
1. 确定检查范围（哪些表/字段） → 
2. 逐维度执行检查并记录结果 →
3. 计算各维度得分（0-100） →
4. 生成质量报告 →
5. 标记不通过项 → 决定操作：跳过 / 重新清洗 / 中止
```

## 质量评分方法

> 对应 核心步骤 第 3 步

| 维度 | 评分公式 | 权重 |
|------|----------|------|
| 完整性 | (1 - NULL率) × 100 | 25% |
| 一致性 | (1 - 违反率) × 100 | 25% |
| 准确性 | (1 - 超界率) × 100 | 20% |
| 时效性 | min(延迟/SLA, 1) × 100 取补 | 10% |
| 唯一性 | (1 - 重复率) × 100 | 10% |
| 有效性 | (1 - 无效率) × 100 | 10% |

**总分 = 各维度得分 × 权重之和**

| 总分 | 评级 | 建议操作 |
|------|------|----------|
| ≥ 95 | 优秀 | 可直接用于分析 |
| 85 - 94 | 良好 | 注意低分维度 |
| 70 - 84 | 一般 | 建议针对性修复后使用 |
| < 70 | 不合格 | 必须重新清洗或拒绝使用 |

## 集成方式

| 上游 | 下游 |
|------|------|
| data-clean（清洗后触发） | data-analysis（质量达标才进入分析）|
| data-query（首次取数后） | model（建模前质量门禁） |
| context-retriever（多源融合前） | report（报告中包含质量说明） |

## 注意事项

- 仅有统计数据时不运行质量检查（不够数据量）
- 质量检查只读不写，不修改数据
- 检查范围过大时抽样检查（>100万行按1%抽样）
- 复合检查项按需启用，不默认全部执行

## 常见借口与纠正

| 借口 | 现实 |
|--------|---------|
| "数据刚清洗过，质量肯定没问题" | 清洗不一定覆盖所有质量维度；完整性/一致性/准确性可能在不同步骤中引入新问题 |
| "跑全量检查太慢了，抽几个字段看看就行" | 抽样可能遗漏关键字段的质量缺陷；明确抽样策略并记录在报告中 |
| "分数 84 和 85 差不多" | 阈值是硬标准：<85 必须修复后使用，不可模糊通过 |

## 红线

- **P0 维度零分:** 任一关键维度（如主键唯一性）得分为 0 — 数据不可使用
- **准确性违规 >5%:** 准确性维度违规率超 5% — 数据可能来自错误来源
- **时效性 SLA 翻倍:** 数据新鲜度延迟超 2× SLA — 数据可能过时且产生误导
- **仅对汇总数据检查:** 质量检查仅基于统计汇总而无明细行 — 结果无意义

## 验证

1. 验证检查范围已定义：确认表、字段和抽样策略已明确记录
2. 验证全维度已执行：确认 6 个维度（完整性/一致性/准确性/时效性/唯一性/有效性）均有记录结果
3. 验证评分正确：确认各维度得分使用正确公式和权重
4. 验证失败项已标记：确认每个失败项列出了具体字段和问题描述
5. 验证总分正确：确认总评级正确反映加权维度得分
6. 验证建议已给出：确认建议字段给出了明确的操作指令
